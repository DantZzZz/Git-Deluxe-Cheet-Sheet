01 Setup & Configuration
First-time Git users: set your identity before making any commits. This info is baked into every commit you make.
$ git config --global user.name "Your Name" setup +
Sets your name globally for all Git repos on this machine. This appears in every commit you make.
Example
# Set identity (one-time setup) git config --global user.name "Ada Lovelace" git config --global user.email "ada@example.com" # Verify your config git config --list
$ git init setup +
Creates a new Git repository in the current folder. A hidden .git/ directory is created to track history.
Example
# Create a new project and init git mkdir my-project && cd my-project git init # Output: Initialized empty Git repository in .git/
$ git clone <url> setup +
Downloads a complete copy of a remote repository (all files, history, branches) to your local machine.
Example
# Clone via HTTPS git clone https://github.com/user/repo.git # Clone into a custom folder name git clone https://github.com/user/repo.git my-folder # Clone a specific branch only git clone --branch develop https://github.com/user/repo.git
$ git config --global core.editor "code --wait" setup +
Sets VS Code as your default editor for Git operations (commit messages, rebase, etc). Replace with vim, nano, etc.
Useful config options
# Set VS Code as editor git config --global core.editor "code --wait" # Default branch name (modern: main) git config --global init.defaultBranch main # Handy alias: git st → git status git config --global alias.st status
02 Inspecting & Navigating
$ git status inspect +
Your most-used command. Shows: which branch you're on, what's staged for commit, what's modified but not staged, and untracked files.
Example
git status # Short/compact view git status -s # M = modified, A = added, ?? = untracked
$ git log --oneline --graph --all inspect +
Shows the commit history. --oneline condenses output, --graph draws branch structure, --all shows all branches.
Useful variants
# Compact history git log --oneline # Visual branch graph git log --oneline --graph --all # Filter by author git log --author="Ada" # Show last 5 commits git log -5
$ git diff inspect +
Shows line-by-line differences. By default shows unstaged changes. Use --staged to see what's about to be committed.
Variants
# Unstaged changes (working dir vs index) git diff # Staged changes (what will be committed) git diff --staged # Compare two branches git diff main..feature-branch # Diff a specific file git diff src/app.js
$ git show <commit-hash> inspect +
Displays details of a specific commit: metadata (author, date, message) and the full diff of changes introduced.
Example
# Show the latest commit git show # Show a specific commit by hash git show a1b2c3d # Show only files changed in that commit git show --name-only a1b2c3d
$ git blame <file> inspect +
Annotates every line in a file showing who last changed it and in which commit. Useful for tracking down when a bug was introduced.
Example
git blame src/utils.js # Show specific line range (lines 40-60) git blame -L 40,60 src/utils.js
$ git grep "search term" inspect +
Searches through tracked files in the working directory (much faster than regular grep on large repos).
Example
# Find "TODO" in all tracked files git grep "TODO" # Show line numbers git grep -n "useEffect" # Search only .js files git grep "fetch" -- "*.js"
03 Staging & Committing
git add moves changes to the staging area (index). git commit permanently records staged changes into history. Think of staging as preparing a package before shipping it.
$ git add . stage +
Stages all changes in the current directory (new, modified, deleted). Use -p for interactive hunk-by-hunk staging.
Variants
# Stage everything git add . # Stage a specific file git add src/index.js # Interactively pick hunks to stage git add -p # Stage all .css files git add "*.css"
$ git commit -m "feat: add login" commit +
Records staged changes as a new commit. Write clear, imperative messages: "Add feature" not "Added feature". Follow Conventional Commits for team projects.
Examples
# Standard commit git commit -m "feat: add dark mode toggle" # Stage tracked files + commit in one step git commit -am "fix: correct typo in header" # Amend last commit message git commit --amend -m "corrected message" # Open editor for multi-line message git commit
$ git restore --staged <file> stage +
Removes a file from the staging area (unstages it) without discarding the actual changes in the file. The modern replacement for git reset HEAD <file>.
Examples
# Unstage a specific file git restore --staged config.js # Discard changes in working directory (careful!) git restore config.js # Discard all unstaged changes git restore .
$ git stash push -m "wip: feature" stash +
Temporarily shelves uncommitted changes so you can switch context. Think of it as a clipboard for your work-in-progress.
Stash workflow
# Save current changes to stash git stash # Stash with a descriptive name git stash push -m "wip: navbar redesign" # List all stashes git stash list # Apply most recent stash (keep it) git stash apply # Apply and remove from stash list git stash pop # Delete a specific stash git stash drop stash@{1}
$ echo "node_modules/" >> .gitignore ignore +
The .gitignore file tells Git which files/folders to never track. Essential for build artifacts, secrets, and dependencies.
Common .gitignore patterns
# Dependencies node_modules/ vendor/ # Environment / secrets .env .env.local # Build output dist/ build/ *.min.js # OS files .DS_Store Thumbs.db # Remove already-tracked file git rm --cached secrets.txt
04 Branching & Merging
Branch early, branch often. Branches are cheap in Git. Use a new branch for every feature, bug fix, or experiment. Never commit directly to main.
$ git branch branch +
Lists all local branches. The current branch is marked with *. Use -a to show remote branches too.
Branch management
# List local branches git branch # List all (local + remote) git branch -a # Create a new branch git branch feature/login # Delete a merged branch git branch -d feature/login # Force delete (even if unmerged) git branch -D experiment
$ git switch -c feature/auth branch +
Creates and switches to a new branch in one command. git switch is the modern, safer replacement for git checkout for branch operations.
Examples
# Create + switch (modern way) git switch -c feature/auth # Classic equivalent git checkout -b feature/auth # Switch to existing branch git switch main # Return to previous branch git switch -
$ git merge feature/auth merge +
Integrates changes from another branch into the current branch. Creates a "merge commit" if histories have diverged. Always merge INTO the destination branch.
Merge workflow
# Switch to the target branch first git switch main # Merge the feature branch in git merge feature/auth # No-fast-forward (always create merge commit) git merge --no-ff feature/auth # Abort a merge in progress git merge --abort
$ git rebase main rebase +
Re-applies your branch's commits on top of another branch, creating a linear history. Cleaner than merging but never rebase shared/public branches.
Rebase workflow
# Rebase your feature branch onto main git switch feature/auth git rebase main # Interactive rebase: squash/edit last 3 commits git rebase -i HEAD~3 # Continue after resolving conflicts git rebase --continue # Bail out if things go wrong git rebase --abort
$ git cherry-pick a1b2c3d branch +
Applies the changes from a specific commit onto the current branch. Useful for pulling a hot-fix from one branch to another without merging everything.
Example
# Apply a single commit by hash git cherry-pick a1b2c3d # Apply a range of commits git cherry-pick a1b2c3d..f4e5d6c # Cherry-pick without committing (stage only) git cherry-pick -n a1b2c3d
05 Remote Sync
$ git remote add origin <url> remote +
Links your local repo to a remote repository. origin is the conventional name for your primary remote (usually GitHub).
Remote management
# Add a remote called "origin" git remote add origin https://github.com/user/repo.git # List remotes and their URLs git remote -v # Change a remote's URL git remote set-url origin https://github.com/user/new-repo.git # Remove a remote git remote remove origin
$ git push -u origin main push +
Sends your local commits to the remote. -u sets the upstream tracking so future pushes just need git push.
Push variants
# First push (set upstream) git push -u origin main # Subsequent pushes git push # Push all branches git push --all # Push a specific branch git push origin feature/auth # Delete a remote branch git push origin --delete old-branch
$ git pull --rebase pull +
git pull = git fetch + git merge. Using --rebase keeps history linear (preferred by many teams). Fetching first lets you review before merging.
Pull strategies
# Pull with merge (default) git pull # Pull with rebase (cleaner history) git pull --rebase # Fetch only (don't merge yet) git fetch origin # See what's coming before merging git log HEAD..origin/main --oneline
$ git fetch --prune fetch +
Downloads remote changes to your local cache but doesn't touch your working directory. --prune removes stale remote-tracking branches that no longer exist on remote.
Example
# Fetch and clean stale branches git fetch --prune # Fetch all remotes git fetch --all # Fetch a specific branch git fetch origin main
06 Undoing & Fixing Mistakes
⚠️ Commands marked DANGER rewrite history. Only use them on local or private branches — never on shared branches that others have pulled.
$ git revert HEAD safe undo +
Creates a NEW commit that undoes the changes from a previous commit. The safest way to undo — it doesn't rewrite history, so it's safe for shared branches.
Example
# Revert the most recent commit git revert HEAD # Revert a specific commit git revert a1b2c3d # Revert without auto-committing git revert --no-commit HEAD
$ git reset --soft HEAD~1 rewrite +
Moves the branch pointer back. Three modes: --soft keeps changes staged, --mixed (default) unstages them, --hard ⚠️ permanently discards them.
Reset modes
# Undo last commit, keep changes staged git reset --soft HEAD~1 # Undo last commit, unstage changes (default) git reset HEAD~1 # ⚠️ DANGER: discard last commit + changes git reset --hard HEAD~1 # ⚠️ DANGER: match remote exactly (wipe local) git reset --hard origin/main
$ git commit --amend rewrite +
Modifies the most recent commit — can change the message, add forgotten files, or both. ⚠️ Rewrites history, so only use before pushing.
Example
# Fix the commit message git commit --amend -m "feat: better message" # Add a forgotten file to last commit git add forgotten.js git commit --amend --no-edit
$ git reflog recover +
Your safety net! Git's reflog records every position your HEAD has been. If you accidentally lose commits with a hard reset, use this to find and recover them.
Recovery workflow
# See full movement history of HEAD git reflog # Restore to a previous state (e.g. before bad reset) git reset --hard HEAD@{2} # Recover a "deleted" branch git checkout -b recovered a1b2c3d
$ git clean -fd clean +
⚠️ Permanently removes untracked files and directories. Run with -n (dry run) first to preview what will be deleted.
Example
# Dry run — preview what would be deleted git clean -nfd # Actually delete untracked files + dirs git clean -fd # Also remove gitignored files git clean -fdx
07 Tags & Releases
$ git tag -a v1.0.0 -m "Release 1.0" tags +
Tags mark specific commits as release points. Annotated tags (-a) store extra metadata (tagger, date, message) and are preferred for releases. Use semantic versioning: vMAJOR.MINOR.PATCH.
Tag workflow
# Create an annotated tag git tag -a v1.0.0 -m "Initial release" # List all tags git tag # Push tags to remote git push origin --tags # Delete a tag locally + remotely git tag -d v1.0.0 git push origin --delete v1.0.0
08 GitHub Collaboration
GitHub builds on Git with Pull Requests, Issues, Actions, and more. The standard collaboration flow: Fork → Clone → Branch → Push → Pull Request → Review → Merge.
⇒ Pull Request (PR) GitHub +
A PR is a request to merge your branch into another (usually main). It enables code review, discussion, and CI checks before changes land. Open one via GitHub UI or the gh CLI.
PR via GitHub CLI
# Push branch and open PR in browser git push -u origin feature/auth # Using GitHub CLI (gh) gh pr create --title "Add auth" --body "Closes #42" # Check PR status gh pr status # Check out someone else's PR locally gh pr checkout 123
⇒ Fork & Upstream Sync GitHub +
Forking creates your own copy of a repo. To contribute to open source: fork → clone YOUR fork → add the original as upstream → keep in sync.
Fork sync workflow
# After forking on GitHub, clone your fork git clone https://github.com/YOU/repo.git # Add the original repo as upstream git remote add upstream https://github.com/ORIGINAL/repo.git # Sync your fork with upstream git fetch upstream git merge upstream/main
⇒ GitHub Actions CI/CD +
GitHub Actions automates workflows triggered by Git events (push, PR open, release). Define workflows in YAML files under .github/workflows/.
Basic CI workflow .yml
name: CI on: [push, pull_request] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm install - run: npm test
⇒ Linking Issues in Commits GitHub +
Use keywords in commit messages or PR descriptions to automatically close GitHub Issues when merged into the default branch.
Keywords
# In commit message or PR body: git commit -m "fix: resolve null pointer — Closes #42" # Keywords: Closes, Fixes, Resolves # Works with: #42, org/repo#42, full URL
$ ssh-keygen -t ed25519 setup +
SSH keys let you authenticate with GitHub without entering your password. Generate a key, add the public key to GitHub Settings → SSH Keys.
SSH setup
# Generate SSH key ssh-keygen -t ed25519 -C "you@email.com" # Copy public key to clipboard (macOS) pbcopy < ~/.ssh/id_ed25519.pub # Test connection ssh -T git@github.com
09 Power Tools & Shortcuts
$ git bisect start debug +
Uses binary search through your commit history to find which commit introduced a bug. Dramatically faster than manual searching on large repos.
Bisect session
git bisect start git bisect bad # current commit is broken git bisect good v2.1.0 # this version was fine # Git checks out midpoint — test it, then: git bisect good # or bad # Repeat until the culprit is found git bisect reset # return to HEAD
$ git worktree add ../hotfix main advanced +
Check out multiple branches simultaneously in separate directories from the same repo. Useful for working on a hotfix without stashing your current work.
Example
# Create a linked working tree for hotfix git worktree add ../hotfix-work main # List all worktrees git worktree list # Remove when done git worktree remove ../hotfix-work
$ git shortlog -sn inspect +
Summarizes commit counts by author. Great for seeing who contributed what to a project.
Example
# Sorted by number of commits git shortlog -sn # Include all branches git shortlog -sn --all
$ git submodule add <url> advanced +
Embeds another Git repository inside your repo at a specific commit. Useful for managing shared libraries or dependencies as separate repos.
Submodule basics
# Add a submodule git submodule add https://github.com/org/lib.git libs/lib # Clone repo WITH all submodules git clone --recurse-submodules <url> # Update all submodules git submodule update --remote --merge
